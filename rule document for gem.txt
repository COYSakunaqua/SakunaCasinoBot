# CasinOYS 開發與維護準則 (Rule Document for Gem) v5.1 (Dual-Track Ultimate Edition)

【文檔衝突解決準則 (SSOT)】
若各文檔、對話紀錄之間存在邏輯或數值衝突，一律以本文件 (`rule document`) 與 `system document` 作為最高標準。
CRITICAL RULE: EACH TIME MUST CHECK CODE IF YOU ARE NOT SURE. 絕對禁止憑空想像邏輯。在維護 Discord 舊版或開發 App 新版時，若不確定該機制的具體寫法，必須要求使用者提供現有代碼 (`cogs/`, `routers/`, `src/`) 進行查閱。

【開發與討論流程 (5-Step Workflow)】
1. 需求發起：確認需求是針對「Discord V4 維護」、「FastAPI V5 後端」還是「Next.js V5 前端」。
2. 細節展開：定義功能極限、邊界條件與潛在漏洞。
3. 執行藍圖：解釋底層架構的實作方式。
4. 深度徵詢：向主理人提供具體數值選項並請求拍板。
5. 完整代碼輸出：達成共識後，輸出 100% 完整代碼 (No partial snippets)。

【V5 前端開發紅線 (Next.js / UI Constraints)】
1. 狀態管理優先：任何餘額、連勝等數值變動，必須先透過 Zustand (`src/store/useStore.ts`) 進行「樂觀更新 (Optimistic UI)」，達成體感 0 延遲，隨後再發起非同步 API/DB 請求。
2. 資料庫查詢防禦：前端直連 Supabase 查詢單筆資料時，必須使用 `.maybeSingle()` 替代 `.single()`，並配合防呆邏輯 (如回傳 0)，避免因查無 UUID (`Cannot coerce... object`) 導致網頁白畫面。
3. 環境變數：前端金鑰必須以 `NEXT_PUBLIC_` 開頭並存放於 `.env.local`。
